## ssh_key / headless sync

- **Why `GIT_SSH_COMMAND`, not `core.sshCommand`:** the per-repo `ssh_key` is injected as an env var on the git child process (`internal/git/git.go`, `runCtx`), deliberately *not* written into each repo's git config. keeps sexton non-invasive (it never mutates the synced repos) and keeps the key path in the fleet's sexton YAML. don't "consolidate" this into `git config core.sshCommand`.

- **Env override depends on append order:** `runCtx` does `append(os.Environ(), "GIT_SSH_COMMAND="+...)` — appended last so it wins over any ambient `GIT_SSH_COMMAND`. with no `ssh_key`, `cmd.Env` stays nil and git inherits the ambient value (the user's `ssh-agent`). both paths are pinned by tests in `git_test.go`; preserve last-wins if you refactor env handling.

- **Scoped out on purpose:** no `git_ssh_command` full-override field, and no host-key / `StrictHostKeyChecking` / port options. host keys come from the user's `~/.ssh/known_hosts`; a fresh `systemctl --user` env that has never connected interactively will fail host verification until the host is known. add options only if that actually bites.

- **Verified live (2026-06-19):** the updated sexton is running as a `systemctl --user` service on this host, syncing a real SSH remote with a passphrase-less deploy key — the headless auth path is confirmed end-to-end, not just against the fake-git harness.
